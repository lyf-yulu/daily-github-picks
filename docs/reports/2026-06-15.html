idea-to-ship ⭐ 55

从想法到上线的可组合 AI 智能体技能套件

Claude Agent SkillsOpenAI CodexMarkdown 工作流文档多智能体编排
idea-to-ship 是一套可组合的 Agent Skills（Agent 技能集），同时适配 Claude 与 OpenAI Codex，专为独立开发者和小团队设计，用于补齐 AI 编码工作流中常被跳过的前半程：决定什么值得做、诚实验证、规划可分步交付的构建。整套流程包含四个核心技能：ideate（将模糊想法锁定为带成功指标和淘汰标准的 CONCEPT_BRIEF）、deep-dive（多智能体并行调研加红队对抗式审查）、prompt-pack（把概念拆成自包含、可独立交付的构建提示）、build-loop（驱动构建到接近成品的工艺水准，含渲染、截图、自我评测多轮迭代）。此外还提供 autopilot 自动驾驶模式，可无人值守地串联整条流水线。各技能既能独立使用，也能链式编排，并附有 Redline 案例研究展示一次 13.5 小时通宵自动构建的真实产出。

Architect-Loop 跨厂商智能体循环 ⭐ 387

Claude 设计、Codex 实现、仓库即记忆

Claude Code SkillAnthropic ClaudeOpenAI CodexGit
Architect-Loop 是一个面向 Claude Code 的研究型 Skill，旨在构建一种跨厂商的双智能体协作循环。它将不同厂商的大模型按照各自所长进行角色分工：Claude（项目中称为 Fable 5）担任架构师，负责需求拆解、方案设计与代码评审；GPT-5.5 Codex 担任建造者，负责具体的代码实现与编辑落地；而 Git 仓库本身则被当作两者之间的共享长期记忆，所有上下文、决策与中间产物都通过文件系统与提交历史进行沉淀和交换。这种设计回避了单一模型既当设计者又当执行者时容易出现的自我确认偏差，让规划与实现在不同模型间形成相互制衡的反馈环。该项目以 Claude Code Skill 的形式分发，开发者只需在本地启用 Skill，即可在熟悉的 Claude Code 工作流中触发架构师与建造者的轮转协作，把仓库变为可审计、可回溯的协同工作空间。其底层依赖 Claude Code CLI、OpenAI Codex 接口以及标准 Git 工作流，强调研究背书与可复现的提示工程实践，适合希望系统性探索多模型协同与长期记忆机制的工程师与研究者参考。
在多数 Agent 框架仍局限于单一模型自循环的当下，该项目以研究为依据，明确把架构与实现拆给不同厂商的模型，并用仓库充当长期记忆，提供了一种轻量、可落地的跨厂商协作范式，对探索多模型工程化协同有参考价值。

Loop Engineering 工具集 ⭐ 179

为 AI 编码代理设计循环工程模式与 CLI

CLI 工具AI Coding AgentsPrompt/Loop OrchestrationMarkdown 模式文档
Loop Engineering 是一个面向 AI 编码代理（Coding Agents）的循环工程实践仓库，由 Cobus Greyling 维护，灵感源自 Addy Osmani 与 Boris Cherny 等业界专家关于代理式编程的思想。项目核心定位是把传统的「提示工程」升华为「循环工程」，即不再只是单轮提示模型，而是设计一整套能够持续提示、调度、审计 AI 代理的系统化工作流。仓库提供了可直接复用的实践模式、入门脚手架（starters）以及三款 CLI 工具：loop-audit 用于审计代理在循环执行中的行为表现与质量，loop-init 用于快速初始化一个符合循环工程范式的项目骨架，loop-cost 则用于跟踪与估算多轮代理调用的 Token 与费用开销。这套工具试图回答工程团队在落地 AI Agent 时常遇的问题：如何保证多轮自主决策的可控性、可观测性与成本透明。通过将循环、审计、成本三个维度封装为标准化命令行，开发者可以在自有代码库中快速搭建可重复、可演进的 Agent 编排管线，而不必从零设计 reasoning loop、日志格式与计费策略。整体技术栈以脚本化 CLI 为主，强调与现有 AI 编码代理生态（如 Claude Code、Cursor、Aider 等）的协作集成。
随着 Claude Code、Cursor 等代理式编程工具走向主流，单纯的 Prompt Engineering 已无法满足生产需求，开发者更需要面向「循环」的系统设计方法论。该仓库较早把循环工程具象化为可执行的 CLI 与模式集合，并直接嵌入审计与成本管理两个工程刚需，为团队从玩具 Demo 过渡到可治理的 Agent 工作流提供了参考蓝本，值得关注 Agent 工程化方向的开发者借鉴。

SkillRevise 是一个面向大语言模型（LLM）智能体的技能修订框架，专注于通过执行轨迹（trace）来条件化地优化和修订由 LLM 自动生成的智能体技能。该项目的核心定位是解决 LLM 自主生成技能时常见的质量不稳定、错误累积和适应性不足问题，通过引入轨迹条件化机制，让模型能够基于实际运行反馈对已有技能进行迭代式改进。项目作为官方发布版本提供研究代码与实验复现入口，适用于智能体技能学习、自我改进及自动化工作流构建等场景。由于仓库目前仅包含简短描述，详细的功能模块、API 接口和实验脚本仍待项目方进一步公开。技术栈方面，作为典型的 LLM 智能体研究项目，预计基于 Python 生态，结合主流 LLM 调用框架与提示工程工具链实现技能生成、轨迹采集与修订循环。

Omnigent 元代理框架 ⭐ 1,200

统一管理多家 AI 编码代理的元框架

Python 3.12+uv / pip / Homebrew 分发Node.js 22 LTStmuxWeb UI (localhost:6767)macOS 桌面应用Modal / Daytona 云沙箱Claude Code / Codex / Pi CLIDatabricks SDK
Omnigent 是一个面向 AI 代理的「元 harness（元运行壳）」，旨在为 Claude Code、Codex、Pi 以及用户自定义代理提供统一的运行与协作层。开发者无需为每种代理重写适配逻辑，即可在同一会话中切换或组合不同的 harness，并通过策略与沙箱机制对其进行约束。项目的核心定位是让多代理协作变得像调用一个工具那样简单：会话可以从终端开始，无缝切换到浏览器或手机端，消息、子代理、终端与文件状态实时同步。功能亮点上，Omnigent 支持多代理监督模式，例如让一个代理评审另一个代理的产出，或将任务拆分给各自擅长的代理；模型层面兼容 API Key、Claude/ChatGPT 订阅、OpenAI/Anthropic 兼容网关（OpenRouter、LiteLLM、Ollama、vLLM、Azure）以及 Databricks 工作区。它还内置了基于 Modal 与 Daytona 的云沙箱执行能力，可在一次性环境中运行会话，并提供策略系统在敏感操作前请求审批、限制花费或限制工具调用范围。仓库附带两个示例代理：Polly 作为多代理编码协调员在并行 git worktree 中分派任务并跨厂商交叉评审，Debby 则同时调用 Claude 与 GPT 双头脑进行头脑风暴与辩论。技术栈方面，项目基于 Python 3.12+，通过 uv/pip/Homebrew 分发，依赖 Node.js 用于驱动各家代理 CLI，并使用 tmux 包装原生终端会话，配套本地 Web UI 与 macOS 桌面应用。
在 AI 编码代理生态高度碎片化的当下，Omnigent 提出了「元 harness」这一新颖抽象：它不与现有代理竞争，而是将 Claude Code、Codex、Pi 等放在同一治理层下协同工作，并通过沙箱与策略解决多代理时代的安全与成本失控问题。其跨设备实时协作、跨厂商交叉评审（如 Polly）以及双模型并行辩论（如 Debby）的设计模式，为多代理工程实践提供了可复用范式，对希望在团队中规模化使用 AI 代理的开发者尤具参考价值。
